## Project Overview

TraxJob is a job-application tracker that helps job seekers log applications,
move them through stages (Wishlist → Applied → Interview → Offer / Rejected),
search / filter / sort, and export. The frontend is a Vite + React SPA. A
**standalone Node/Express + PostgreSQL (Drizzle ORM) backend** (`server/` and
`db/`) provides authentication (cookie sessions, argon2) and will back data
persistence. Jobs currently still use the browser localStorage repository on the
client; auth is server-backed.

No Rails, no ERB, no Alpine/Stimulus, no Next.js/SSR.

## Tech Stack

- Vite 6 — dev server + build (`npm run dev` / `npm run build`).
- React 19 — required. shadcn/ReUI components use ref-as-prop; React 18 breaks
  `asChild` triggers.
- TypeScript 5 (strict).
- Tailwind CSS v4 — Material You (MD3) tokens mapped to utilities in
  `src/styles/theme.css` (see `design.md`). The `md-*` utilities are the single
  source of truth for color / radius / shadow, and re-resolve in dark mode.
- shadcn/ui (`src/components/ui/`) + ReUI (`src/components/reui/`) components.
- React Router 7 — `/` landing, `/app` tracker; pages in `src/pages/`.
- Persistence: `JobsRepository` interface in `src/storage/` (localStorage today,
  cloud-ready). The UI goes through the `useJobs` hook, never localStorage.
- Node 22 required.

Backend (`server/`, `db/`):
- Node + Express 4 (run with `tsx`), PostgreSQL 16 + Drizzle ORM (schema in
  `db/schema.ts`).
- Auth: cookie sessions (hashed token in `sessions` table), `@node-rs/argon2`
  for password hashing, `nodemailer` (Gmail) for password-reset email, `zod`
  for request validation.
- Env is validated in `server/env.ts`; never read `process.env` ad hoc in
  handlers. Validate request bodies with zod. Never log secrets or password
  hashes. React rules below do NOT apply to `server/` (no JSX, no localStorage
  rule); the JS/TS rules do.

## JavaScript / TypeScript Rules

Applies to all `.js`, `.ts`, `.jsx`, and `.tsx` files.

- **Quotes**: Double quotes for strings. Single quotes only for apostrophes. Template literals only for interpolation.
- **Semicolons**: Always required.
- **Indentation**: 2 spaces, never tabs.
- **Space after keywords**: `if (condition)` not `if(condition)`.
- **Space before blocks**: `if (admin) {` not `if (admin){`.
- **Strict equality**: Always `===` / `!==`, never `==` / `!=`.
- **String interpolation**: Use template literals, never `+` concatenation.
- **Naming**: camelCase for variables and functions, PascalCase for React components and types. No snake_case.
- **Arrays**: Use `[]` literals, never `new Array()`.
- **Object key spacing**: `{ key: "value" }` — space after colon only.
- **Commas**: Space after: `[1, 2, 3]` not `[1,2,3]`.
- **else placement**: Same line as closing brace: `} else {`.
- **Curly braces**: Required for multi-line `if` bodies.
- **Ternary (multi-line)**: `?` and `:` on their own lines, indented.
- **Ternary in strings**: If only part of a string differs, use a template literal with the ternary inside — don't repeat the full string in both branches.
- **Block padding**: No blank lines at start/end of blocks.
- **Files**: Must end with a newline.
- **Comments**: Keep comments minimal — **delete non-critical comments**. Remove comments that merely restate what the code already says (e.g. `/* UI state */`, `/* loop over rows */`, `/* set the value */`) or that narrate obvious steps. Keep only comments that explain non-obvious *why* (intent, edge cases, gotchas). Any comment that remains must use `/* ... */`, not `//`.
- **Error logging**: Must include a feature-name prefix, e.g. `console.error("[TraxJob] message")`.
- **Prefer `const`**: Use `const` by default, `let` only when reassigned, never `var`.
- **Types**: Prefer explicit types on function signatures and exported APIs. Avoid `any` — use `unknown` plus narrowing if the type is genuinely unknown.

## React Rules

Applies to `.tsx` components and `src/pages/` files.

- **Client SPA**: Plain React function components with hooks. There are no
  Server Components and no `'use client'` directives — this is not a Next.js app.
- **Persistence boundary**: Read and mutate jobs only through the `useJobs` hook
  / `JobsRepository` (`src/storage/`). Never call `localStorage` directly from
  components.
- **Styling via tokens**: Use the `md-*` Tailwind utilities; never hard-code hex
  colors in components. To restyle globally, change the token in `theme.css`.
  Buttons are pill-shaped (the shadcn `Button` is customized to `rounded-full`).
- **Reuse components**: Prefer the shared components (`@/components/ui/*`,
  `@/components/reui/*`, `Button`, `JobModal`, `StatusBadge`, …) over
  re-implementing primitives.
- **Keys**: Stable `key` on list items (not array index when items can reorder).
- **Accessibility**: Associate labels with inputs; icon-only buttons need an
  `aria-label`; decorative elements get `aria-hidden`; respect
  `prefers-reduced-motion`.

## Vendored UI components (out of scope — skip)

**Files under `src/components/ui/` (shadcn/ui) and `src/components/reui/` (ReUI)
are generated by the shadcn CLI** and intentionally follow that library's own
formatting — notably they omit trailing semicolons and use library-specific
conventions. Do NOT flag style issues (e.g. semicolons / JT-B7, comment style)
in these files; skip them entirely. They are regenerated by re-running
`npx shadcn@latest add ...`, so any hand-edits would be overwritten.

## CSS / Tailwind Rules

- Use Tailwind utility classes; avoid ad-hoc inline `style` objects for things
  Tailwind covers.
- Use the MD3 `md-*` tokens for color / radius / shadow — no hard-coded hex.
- Decorative/animated background elements must respect
  `prefers-reduced-motion`.

## Patterns to Avoid

- Non-critical / redundant comments that restate the code — delete them; keep only "why" comments.
- Hard-coded hex colors instead of `md-*` tokens.
- Calling `localStorage` directly from components instead of the repository / `useJobs`.
- Hand-editing vendored shadcn/ReUI files in `src/components/ui` or `src/components/reui`.
- `==` / `!=` comparisons.
- Single-quoted strings (except apostrophes).
- String concatenation with `+`.
- `new Array(...)` constructors.
- `any` types where `unknown` + narrowing would do.
- `var`, and `let` where `const` suffices.
- `//` explanatory comments where `/* ... */` is required.
